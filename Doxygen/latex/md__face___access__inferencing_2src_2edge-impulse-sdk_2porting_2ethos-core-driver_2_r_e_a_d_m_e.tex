This repository contains a device driver for the Arm(\+R) Ethos(\+TM)-\/U NPU.\hypertarget{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md395}{}\doxysection{\texorpdfstring{Building}{Building}}\label{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md395}
The source code comes with a CMake based build system. The driver is expected to be cross compiled for any of the supported Arm Cortex(\+R)-\/M CPUs, which requires the user to configure the build to match their system configuration.

One such requirement is to define the target CPU, normally by setting {\ttfamily CMAKE\+\_\+\+SYSTEM\+\_\+\+PROCESSOR}. {\bfseries{Note}} that when using the toolchain files provided in \href{https://git.mlplatform.org/ml/ethos-u/ethos-u-core-platform.git}{\texttt{ core\+\_\+platform}}, the variable {\ttfamily TARGET\+\_\+\+CPU} must be used instead of {\ttfamily CMAKE\+\_\+\+SYSTEM\+\_\+\+PROCESSOR}.

Target CPU is specified on the form "{}cortex-\/m$<$nr$>$$<$features$>$"{}, for example\+: "{}cortex-\/m55+nodsp+nofp"{}.

Similarly the target NPU configuration is controlled by setting {\ttfamily ETHOSU\+\_\+\+TARGET\+\_\+\+NPU\+\_\+\+CONFIG}, for example "{}ethos-\/u55-\/128"{}.

The build configuration can be defined either in the toolchain file or by passing options on the command line.

Note that if {\ttfamily ethosu\+\_\+wait} is invoked from a different thread and concurrently with {\ttfamily ethosu\+\_\+invoke\+\_\+async}, the user is responsible to guarantee that {\ttfamily ethosu\+\_\+wait} is called after a successful completion of {\ttfamily ethosu\+\_\+invoke\+\_\+async}. Otherwise {\ttfamily ethosu\+\_\+wait} might fail and not actually wait for the inference completion.\hypertarget{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md399}{}\doxysubsection{\texorpdfstring{Driver initialization}{Driver initialization}}\label{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md399}
In order to use a driver it first needs to be initialized by calling the {\ttfamily init} function, which will also register the handle in the list of available drivers. A driver can be torn down by using the {\ttfamily deinit} function, which also removes the driver from the list.

The correct mapping is one driver per NPU device. Note that the NPUs must have the same configuration, indeed the NPU configuration can be only one, which is defined at compile time.\hypertarget{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md400}{}\doxysection{\texorpdfstring{Implementation design}{Implementation design}}\label{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md400}
The driver is structured in two main parts\+: the driver, which is responsible to provide an unified API to the user; and the device part, which deals with the details at the hardware level.

In order to do its task the driver needs a device implementation. There could be multiple device implementation for different hardware model and/or configurations. Note that the driver can be compiled to target only one NPU configuration by specializing the device part at compile time.\hypertarget{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md401}{}\doxysection{\texorpdfstring{Data caching}{Data caching}}\label{md__face___access__inferencing_2src_2edge-impulse-sdk_2porting_2ethos-core-driver_2_r_e_a_d_m_e_autotoc_md401}

To ensure the correct functionality of the driver mutexes and semaphores are used internally. The default implementations of mutexes and semaphores are designed for a single-\/threaded baremetal environment. Hence for integration in environemnts where multi-\/threading is possible, e.\+g., RTOS, the user is responsible to provide implementation for mutexes and semaphores to be used by the driver.

The mutex and semaphore APIs are defined as weak linked functions that can be overridden by the user. The APIs are the usual ones and described below\+:
